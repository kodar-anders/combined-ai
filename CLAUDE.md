## What this is

A **plain TypeScript library** meant to be imported by other projects. Its purpose is to **combine several AI services/providers into one package** â€” selecting a single provider for a prompt, and (later) combining multiple providers on one prompt. `src/index.ts` is the public entry point â€” re-export the package's public API from there. The foundations (build, test, lint, type-check) are in place and three providers (Anthropic, OpenAI, Gemini) are implemented.

### Library structure (current)

- `src/types.ts` â€” the provider-agnostic contract every provider implements: `Message`, `CompletionRequest`, `CompletionResult`, and `Provider` (`complete()` â†’ full text, `stream()` â†’ `AsyncIterable<string>` of text deltas). **`Message.content` is `string | ContentPart[]`** â€” a bare `string` is sugar for one text part (existing callers/tests untouched). `ContentPart` is a discriminated union: `TextPart` (`{type:"text", text}`), `ImagePart` (`{type:"image", source}`), `FilePart` (`{type:"file", source, filename?}`) â€” image/file added in Phase 1 (multimodal). The widening itself (Phase 0) was the one breaking change to `Message`; **tool parts will still just add members** (additive). `MediaSource` = `{kind:"base64", mediaType, data} | {kind:"url", url, mediaType?}` (base64 carries its MIME type; `url`'s `mediaType` is optional â€” Gemini `fileData` uses it). Each provider maps content per-wire (string passthrough where the API allows it, else mapped parts â€” see provider bullets); **provider support varies and the mapper throws when unsupported** (e.g. OpenAI Chat Completions has no URL file source). When a future `ContentPart` member lands, the per-provider `to*Part`/`to*Block`/`to*Content` mappers and `textOf` in `combine/shared.ts` (which filters to text parts to render the prompt â€” combine renders to text, so non-text parts don't propagate) will fail to compile until handled (intended seam). `CompletionRequest.signal?: AbortSignal` is forwarded to every provider `fetch` (in the `init`) for timeout/cancellation; an aborted `fetch` rejects and `providerFetch` wraps it as a **transport** `ProviderError` (`cause` = abort reason). In combine, `completionFor` carries `signal` over (like `model`/`maxTokens`), so one signal threads into every phase/participant call and cancels the whole run. `CompletionResult` also carries `finishReason?: FinishReason` (normalized union `"stop" | "length" | "content_filter" | "other"`), `rawFinishReason?: string` (the provider's exact value), and `refusal?: string` â€” `complete()`-only (no place to surface on `stream()`). Each provider maps its own stop field into the union (see provider bullets); a set `refusal` forces `finishReason` to `"content_filter"`. Lets callers distinguish truncation/refusal from a real empty answer instead of a bare `text: ""`. **Optional + only set when present**, so existing `toEqual({text, model})` tests still pass (Jest ignores `undefined` props). Threads through combine unchanged (`completionFor` passes the result through). New strategy/feature work that constructs a `CompletionResult` should keep these fields. `CompletionResult.usage?: Usage` (`{inputTokens, outputTokens, totalTokens}`) holds token usage â€” each provider's `extractUsage` parses its own field (Anthropic `usage.input_tokens/output_tokens`, no total â‡’ total=in+out; OpenAI `usage.prompt_tokens/completion_tokens/total_tokens`; Gemini `usageMetadata.promptTokenCount/candidatesTokenCount/totalTokenCount` â€” its total **includes thinking tokens** so it's kept verbatim, not in+out). Same optional/only-when-present rule; `complete()`-only. **Structured output:** `CompletionRequest.responseFormat?: ResponseFormat` (`{type:"json_schema", schema, name?}`) â€” a **raw JSON Schema** (no Zod/runtime dep; the user's locked choice). Each provider maps it to its native mechanism (Anthropic `output_config.format`, OpenAI `response_format.json_schema` with `strict:true`, Gemini `responseSchema` via `toGeminiSchema` â€” see provider bullets); applied in `#buildBody`, so it affects `complete()` **and** `stream()`. The model returns JSON in `text`; `complete()` also surfaces `CompletionResult.parsed?: unknown` (the shared `parseStructured` in `src/providers/structured.ts` = `JSON.parse(text)` when `responseFormat` set, else `undefined`; `undefined` too on invalid/truncated JSON â€” raw stays in `text`). **Cross-provider schema requirements** (driven by the strictest providers): every `object` sets `additionalProperties:false` + every prop in `required` with a single **non-null** type; avoid optional/nullable fields (OpenAI strict needs all-required; Gemini wants `nullable:true`, not a `["string","null"]` union or `anyOf`-null â€” `toGeminiSchema` does **not** translate null-unions, so they're not portable), recursive schemas, `$ref`, numeric/length constraints (Gemini ignores most keywords beyond types/enum/format). `complete()`-only for `parsed`; same optional/only-when-present rule.
- `src/errors.ts` â€” the error **vocabulary**: `ProviderError` (exported) + the `apiError` helper (builds the `api` error from a non-2xx `Response`) all three providers use. `apiErrorFromBody(provider, status, data)` is the parallel for a **2xx body that carries `{ error }`** (status kept, e.g. 200) â€” each provider's `complete()` calls it when `isRecord(data.error)`, so a 200-with-error-body throws an `api` `ProviderError` instead of returning `text: ""`. `ProviderError` carries `provider` (`ProviderName`), a `kind` discriminant (`"api"` = error HTTP response â†’ `status` set | `"transport"` = `fetch` itself rejected â†’ `status` undefined, `cause` set), and `code`/`type` parsed from the body (`parseErrorBody`: string `error.code` â†’ `code`; `error.type` or, for Gemini, `error.status` â†’ `type`). Non-2xx message is lowercase `` `${provider} request failed (${status}): ${body}` `` (provider-test assertions follow). **Imports `ProviderName` from `./registry` as `type`-only** so providersâ†’errorsâ†’registry has no runtime cycle â€” keep it type-only. **Mid-stream SSE `error` events + the "no body" case stay plain `Error`** (no HTTP status to attach), outside `ProviderError`'s `api`â‡’`status` invariant. Pure error layer â€” no `fetch`/transport code (that's `transport.ts`); imports nothing from `transport.ts`.
- `src/transport.ts` â€” the HTTP **transport** layer (how a request is made; the error vocabulary is `errors.ts`). Module-internal `providerFetch` wraps the global `fetch` and translates a rejection (network/DNS/abort) into a **transport** `ProviderError`. Exported `requestWithRetry(provider, input, init, retry?)` wraps it with bounded exponential backoff on `RETRYABLE_STATUSES` (429/503/529) for both `complete()`/`stream()`: returns the first non-retryable `Response` (caller still calls `apiError`), honors a `Retry-After` header (seconds or HTTP-date, capped at `MAX_BACKOFF_MS`=60s), and its `sleep` resolves early on `signal` abort so the next `fetch` rejects into the transport error. **Transport rejections are not retried** (scope = HTTP statuses). Config via the per-provider `retry?: RetryOptions` (`{maxRetries?, baseDelayMs?}`, exported from `index.ts`; defaults 2 / 500ms; `maxRetries: 0` disables) â€” each provider stores `#retry` and passes it through. Retry is per-`complete()` call, so combine benefits transparently (no combine changes). Imports `ProviderError` (value) from `./errors` + `ProviderName` (type) from `./registry`; `errors.ts` does **not** import back (one-way). Tested in `src/__tests__/transport.test.ts` (fake timers).
- `src/providers/anthropic.ts` â€” `AnthropicProvider`, talking to the Anthropic Messages API (`POST /v1/messages`) directly over the global `fetch` â€” **no SDK dependency**. Default model `claude-opus-4-8`. Messages map via `toAnthropicMessage` (a `string` content passes through; `ContentPart[]` â†’ content blocks via `toAnthropicBlock`: textâ†’`text`, imageâ†’`image`, fileâ†’`document`; `toAnthropicSource` â†’ `{type:"base64", media_type, data}` or `{type:"url", url}`). Streaming parses the SSE body and yields `content_block_delta` â†’ `text_delta` text. The request body is kept minimal (no `temperature`/`top_p`/`top_k`/`thinking` â€” Opus 4.x rejects sampling params). `finishReason` maps `stop_reason` (`end_turn`/`stop_sequence`â†’`stop`, `max_tokens`â†’`length`, `refusal`â†’`content_filter`, else `other`); `refusal` collects text from `type: "refusal"` content blocks (which `extractText` skips). Structured output â†’ top-level `output_config: {format: {type:"json_schema", schema}}` (schema passed through as-is; no name/strict wrapper).
- `src/providers/openai.ts` â€” `OpenAIProvider`, talking to the OpenAI Chat Completions API (`POST /v1/chat/completions`) directly over `fetch` â€” **no SDK dependency**. Default model `gpt-4.1`. Auth via `Authorization: Bearer`. The optional `system` prompt is folded into a leading `{ role: "system" }` message (OpenAI has no top-level system field), and the token cap is sent as `max_completion_tokens` (works on the o-series too). `toOpenAIMessages` sends a `string` content as-is; `ContentPart[]` â†’ content parts via `toOpenAIPart`: textâ†’`text`, imageâ†’`image_url` (base64 â†’ a `data:` URI, else the url), fileâ†’`file` with `file_data` (base64 only â€” a **URL file source throws**, Chat Completions has none). Streaming parses SSE, yields `choices[0].delta.content`, and stops on the `data: [DONE]` sentinel. `finishReason` maps `choices[0].finish_reason` (`stop`/`length`/`content_filter` pass through, `tool_calls`/`function_call`/elseâ†’`other`); `message.refusal` (when a non-empty string) is surfaced on `refusal` and overrides `finishReason` to `content_filter` even when `finish_reason` is `"stop"`. Structured output â†’ `response_format: {type:"json_schema", json_schema:{name (defaults `"response"`), strict:true, schema}}`.
- `src/providers/gemini.ts` â€” `GeminiProvider`, talking to the Google Generative Language API (`POST /v1beta/models/{model}:generateContent`, streaming via `:streamGenerateContent?alt=sse`) directly over `fetch` â€” **no SDK dependency**. Default model `gemini-2.5-pro`. Auth via the `x-goog-api-key` header. Non-obvious API shape: model + action live in the URL path (not the body); `assistant` role maps to `model`; the optional `system` prompt is a top-level `systemInstruction`; the token cap is `generationConfig.maxOutputTokens`. `toGeminiContent` wraps a `string` content as a single `{text}` part; `ContentPart[]` â†’ parts via `toGeminiPart` (image and file map alike via `toGeminiMedia`: base64 â†’ `inlineData {mimeType, data}`, url â†’ `fileData {mimeType?, fileUri}` â€” `mimeType` omitted when the url source has none). **Gemini `fileData.fileUri` only accepts a Files API / `gs://` URI, not an arbitrary public web URL** (unlike Anthropic/OpenAI image URLs) â€” a plain `https://` url source is rejected by Gemini; documented on `toGeminiMedia`, not translated. **Thinking-token gotcha:** Gemini 2.5 are thinking models whose thinking tokens count against `maxOutputTokens` (unlike Anthropic/OpenAI, where `maxTokens` budgets only visible output), so a tiny cap can return empty/truncated text. We don't disable thinking (the default model can't fully disable it) â€” give Gemini a generous cap. `finishReason` maps `candidates[0].finishReason`, falling back to `promptFeedback.blockReason` when the prompt is blocked (no candidates): `STOP`â†’`stop`, `MAX_TOKENS`â†’`length`, `SAFETY`/`RECITATION`/`BLOCKLIST`/`PROHIBITED_CONTENT`/`SPII`â†’`content_filter`, else `other`. No `refusal` field (Gemini has no refusal-text field â€” the block reason lands in `rawFinishReason`). Structured output â†’ `generationConfig.responseMimeType:"application/json"` + `responseSchema` built by `toGeminiSchema` (recursively UPPERCASEs `type` through `properties`/`items`/`anyOf`/`allOf`/`oneOf`; other keywords pass through; null-unions/`$ref`/constraints not translated).
- `src/registry.ts` â€” `ProviderRegistry`, the package's **single point of access** to its providers. The provider classes are **not exported**; the registry constructs the built-ins by name from a config object. Constructor takes `ProviderRegistryConfig` (`{ anthropic?: AnthropicProviderOptions; openai?: OpenAIProviderOptions; gemini?: GeminiProviderOptions }`) and constructs only the providers present. `select(name)` is typed to `ProviderName` (`"anthropic" | "openai" | "gemini"`) and returns the `Provider` or throws an `Error` listing the configured names; plus `has(name: string)` and `names()` (fixed order: anthropic, openai, gemini). It never reads env vars â€” keys come from the config. Also hosts `combine(request)` (see `src/combine/`): it validates the **shared** request shape (â‰Ą1 participant, unique participants, non-empty `messages`, known `strategy`), resolves participant names â†’ a roster of `{name, provider}` (each via `select()`), and dispatches on `switch (request.strategy ?? "consensus")` with an exhaustiveness `default` that throws. **Consensus-only options** (`minParticipants` an integer in `[1, participants]`, `synthesizer` membership) are validated **inside the consensus branch** via `#validateConsensusOptions` â€” _not_ in the shared preamble, so pipeline (which ignores them) never rejects a valid request over them. The consensus branch also defaults the synthesizer to the first participant. **`responseFormat` is ensemble-only:** rejected (throws) for any non-ensemble strategy right after `strategy` is resolved (it would otherwise be silently dropped by `completionFor`), and in the `ensemble` branch it's **required** (throws if missing) **and must have an object root** (throws on an array/scalar `schema.type` â€” the field-wise vote needs named fields).
- `src/combine/` â€” the **combine** feature: make multiple providers cooperate on one prompt via a selectable strategy. **One file per strategy** (so `combine` doesn't grow into one huge file). Public entry is `ProviderRegistry.combine(request)`.
  - `src/combine/index.ts` â€” the public barrel: the combine types (`StrategyName`, `CombineRequest`, `ParticipantOutcome`, `CombineResult` = `ConsensusResult | PipelineResult | EnsembleResult` discriminated on `strategy`, `EnsembleAgreement`, `CombineUsage`, `CombineEvent` â€” incl. the ensemble `{type:"response"}` variant, `CombineOptions`) plus the one runtime value, `STRATEGY_NAMES` (`consensus`/`pipeline`/`ensemble` â€” the source of truth `combine()` validates against). All result members carry `usage?: CombineUsage` (`{total, byParticipant}`) aggregating every model call the run made. It imports `ProviderName` from `../registry` as a **`type`**, so the combineâ†”registry cycle stays type-only/erasable â€” keep it that way (no value imports from `../registry`).
  - `src/combine/shared.ts` â€” helpers reused by **more than one** strategy: `completionFor`, `composeSystem`, `runOutcome`, `renderConversation`, `sanitizeAnswer` (+ its private `SANITIZE_FRAMING`), `makeEmitter` (wraps `onEvent` into an error-swallowing `emit`), `aggregateUsage`/`outcomeUsage` (+ the `UsageEntry` type), and the `RosterEntry` type. `sanitizeAnswer` returns `{text, usage?}` (not a bare string) so its billed call counts toward the aggregate; both strategies build `usage` by feeding all per-call usages (drafts/critiques/synth/sanitize, or stages/sanitize) through `aggregateUsage`. `completionFor` also carries `request.responseFormat` through (so ensemble's participant calls answer under the schema; consensus/pipeline reject `responseFormat` at the registry, so it's only ever set here for ensemble). A helper lands here only once a 2nd strategy reuses it (the 2â€“3x rule); strategy-private helpers stay in the strategy file.
  - `src/combine/consensus.ts` â€” the **consensus** strategy and its strategy-private helpers (`Survivor`, `LETTERS`, `renderAnswers`, `renderCritiques`, `synthesizerOrder`, `CONCISE_DIRECTIVE`, `CRITIQUE_FRAMING`, `SYNTH_FRAMING`); the cross-strategy helpers live in `shared.ts`. The internal `consensus(roster, synthesizer, request, onEvent?)` returns a `ConsensusResult` and is **not exported from the package**; `registry.ts` imports it from `"./combine/consensus"`. The synthesizer is passed **by name** (a `ProviderName`), not a positional index. Each phase is a shaped `system` + single `user` message to `complete()` â€” no provider-specific code, **no streaming for combine yet**. Three phases:
    - **Phase 1 (draft):** every participant drafts in parallel.
    - **Phase 2 (critique):** every survivor critiques all drafts (**anonymized** to `Answer A/B/C` by default; `attribution: "attributed"` shows provider names â€” the returned `ParticipantOutcome.provider` always carries names regardless), ending with a structured verdict (`BEST:` / `KEY FIX:` / `CONFIDENCE:`).
    - **Phase 3 (synthesize):** the synthesizer reads drafts + critiques and writes the final answer.
    - **Framing invariants â€” preserve if you touch the constants** (rationale in the `consensus.ts` doc-comments): `SYNTH_FRAMING` makes the synthesizer adjudicate on **correctness over popularity** and write a **self-contained answer that never alludes to the drafts/critiques/process or internal labels** (the model kept leaking `Answer A/B/C` / "the candidates" â€” don't reintroduce that). Because prompt-only proved unreliable, a **sanitizing second pass** (`sanitizeAnswer()` + its `SANITIZE_FRAMING`, both in `shared.ts`) rewrites the synthesis to strip leftover process narration, falling back to the un-sanitized answer on empty/failure â€” this is **one extra `complete()` call** on the multi-provider path only. The draft/critique phases prepend `CONCISE_DIRECTIVE`, which strips ceremony **but keeps reasoning/assumptions/caveats** (the audience is other AIs that need the _why_) â€” don't reword it to suppress reasoning, and don't add it to the user-facing synthesis.
    - **Test coupling:** the test's `phaseOf` switches on literal markers in the framing constants â€” `"Rewrite the following"` (sanitize), `"lead assistant"` (synth), `"produce only a critique"` (critique). Keep each constant free of the others' markers.
    - **Events:** `combine()` accepts `CombineOptions` (`{ onEvent? }`) threaded to `consensus`, which emits `CombineEvent`s (`{type:"phase"}` per boundary; `{type:"draft"|"critique", provider, status}` as each call settles) via an `emit` that **swallows handler errors**. Status events only â€” no token streaming, no terminal event; the result is the returned `CombineResult`.
    - **Failure / survivor policy:** fan-outs use `Promise.all` over per-participant `runOutcome()` calls that capture success/failure as a `ParticipantOutcome`, so one provider failing never rejects the round. A successful draft only counts as a survivor if its text is non-empty (mirrors the synthesis empty-guard) â€” an empty `ok` draft is dropped so it can't satisfy `minParticipants` or render as a blank `### Answer`. 0 drafts â†’ throw; single-provider roster â†’ plain completion; fewer than `minParticipants` (default 2) surviving drafts â†’ throw; failed critiques are non-fatal (synthesize from drafts alone); a failed synthesizer falls back to the next survivor.
  - `src/combine/pipeline.ts` â€” the **pipeline** strategy (sequential refinement / conveyor belt). Internal `pipeline(roster, request, onEvent?)` returns a `PipelineResult`; `registry.ts` imports it from `"./combine/pipeline"`. Participant **order is the conveyor order**. Loops the roster keeping a running answer: the first stage with no running answer yet uses `PIPELINE_FIRST_FRAMING`, later stages refine the current answer under `PIPELINE_REFINE_FRAMING` (a `## Question` + `## Current answer` body). A stage **advances** the running answer only if it's `ok` **and** non-empty; failed/empty stages are recorded in `stages` and the previous answer carries forward. Final answer = last advancing stage's text/model/provider. Throws only if **no** stage advanced. **No `minParticipants`/`synthesizer`/`attribution`** (consensus-only â€” ignored). Emits `{type:"stage", provider, status, index}` per stage. **Sanitize** (`sanitizeAnswer`, shared with consensus) runs only when a refining stage actually **changed** the text (tracked via `Running.needsSanitize`) â€” a lone first-stage answer or a verbatim passthrough is returned raw, so the extra call only fires when it can matter. **Framing rationale (preserve if you touch it):** refiners are told to treat the current answer as a strong baseline and preserve correct content (no downstream synthesizer catches a regression). **Test coupling:** `pipeline.test.ts`'s `phaseOf` keys on `"Rewrite the following"` (sanitize, shared), `"revise the current answer"` (refine), else first â€” keep each framing constant free of the others' markers and of consensus's (`"lead assistant"`, `"produce only a critique"`).
  - `src/combine/ensemble.ts` â€” the **ensemble** strategy (multi-model vote on structured output â€” the differentiator). Internal `ensemble(roster, request, onEvent?)` returns an `EnsembleResult`; `registry.ts` imports it from `"./combine/ensemble"`. Every participant answers the prompt **in parallel** under the same `request.responseFormat` (via `completionFor`), each returning a `parsed` object; `runOutcome` captures success/failure. Only `ok` responses whose `parsed` is a **plain object** count (a failed call, empty/invalid JSON â†’ `parsed` undefined, or a non-object top-level value are dropped); throws if **none** qualify. **Mechanical merge** (`mergeObjects`/`mergeField`, no LLM): field-wise over the first-seen union of top-level keys, **every field by majority vote** â€” mode (most common by `stableKey` deep-equality; strict `>` so first-seen wins ties), so the merged value is always one a model actually returned (no synthesized/averaged values; an earlier median was dropped â€” see the plan doc â€” because it fabricated non-schema values and desynced from `agreement`). Per-field `agreement` = `maxCount / objects.length` â€” **denominator is all valid responses**, not just the ones that returned the field, so a field most models omitted scores low (honest confidence). `overall` = mean of per-field. Result carries `merged` (object), `text` (=`JSON.stringify(merged)`), `agreement` (`{overall, byField}`), `responses` (all outcomes, participant order), `usage`. Emits `{type:"response", provider, status}` per settle. **Object-root only:** `registry.combine` rejects an array/scalar-root schema for ensemble up front (the vote needs named fields). **Nested-merge is intentionally shallow** (nested objects/arrays voted as whole values, not recursively merged). **`toSorted` gotcha:** the ES2022 lib lacks it, so `stableKey` uses `.sort()` with an `eslint-disable unicorn/no-array-sort`.
  - **Adding a strategy:** new `src/combine/<strategy>.ts` exporting its orchestrator (returning its own `â€¦Result` member), add a member to the `STRATEGY_NAMES`/`StrategyName` union in `index.ts`, add a `â€¦Result` to the `CombineResult` union (discriminated on `strategy`), and add a `case` to the `switch (request.strategy ?? Promote a helper to `shared.ts` **only when the new strategy actually reuses it** (the 2â€“3x-duplication rule); strategy-private helpers stay in the strategy file. If the strategy emits different progress events, add a `CombineEvent` variant.

They cover the `consensus()` / `pipeline()` / `ensemble()` orchestrators directly. ## Commands

## Architecture / toolchain choices

- **Build = tsup, type-check = tsc.** tsconfig has `noEmit: true`; it exists for editor IntelliSense and `yarn typecheck` only. The build settings live in the `tsup` flags in `package.json`'s `build` script.
- **Tests run through `@swc/jest`** (`jest.config.mjs`), which transpiles TSâ†’CJS fast and does _no_ type-checking â€” types are covered separately by `yarn typecheck`. Test files import globals explicitly from `@jest/globals` (e.g. `import { describe, it, expect } from "@jest/globals"`) rather than relying on ambient types.
- **Dual-format packaging** is wired in `package.json` via the `exports` map (`import`â†’`.js`/`.d.ts`, `require`â†’`.cjs`/`.d.cts`). `files: ["dist"]` and `prepack: yarn build` mean the build is produced on publish/pack.
- **Node globals are typed via `@types/node`** (devDependency), referenced explicitly with `"types": ["node"]` in `tsconfig.json`. This is what makes `fetch`, `Response`, `ReadableStream`, `TextDecoder`, etc. resolve under the ES2022-only `lib` â€” the provider transport relies on them.

## Package management â€” Yarn 4 PnP

Uses **Yarn 4 (Berry) with Plug'n'Play** â€” there is no `node_modules`; deps resolve via `.pnp.cjs` / `.pnp.loader.mjs`. Always use `yarn` (never `npm`).

- **PnP strictness gotcha:** you can only import packages declared directly in `package.json`. A transitive-only dep won't resolve (this is why `@jest/globals` is an explicit devDependency).
- `@jest/globals` versions lag `jest`'s â€” they are not the same number. Match the version Jest actually pulls when bumping.
- `tsconfig` sets `ignoreDeprecations: "6.0"` solely because tsup's `.d.ts` bundler injects a now-deprecated `baseUrl` under TypeScript 6.

## Lint conventions (enforced)

`eslint.config.mjs` is a flat config: type-aware (`strictTypeChecked` + `stylisticTypeChecked`) + unicorn + promise for `**/*.ts`; Jest rules scoped to test files only; a non-type-checked block for `*.{js,cjs,mjs}`. Notable rules:

- **`type` over `interface`**; **`import type`** (inline style) for type-only imports; `consistent-type-exports`.
- Exported API must declare types (`explicit-module-boundary-types`) and functions need return types (`explicit-function-return-type`, callbacks exempt).
- `no-console: error` (a library shouldn't log to a consumer's console) â€” off in tests.
- Prefix intentionally-unused args with `_`; unused imports are an error.
- Single-line `case` clauses omit braces. Prettier owns formatting (ESLint defers via `eslint-config-prettier`).
