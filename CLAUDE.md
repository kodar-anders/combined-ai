## What this is

A **plain TypeScript library** meant to be imported by other projects. Its purpose is to **combine several AI services/providers into one package** ŌĆö selecting a single provider for a prompt, and (later) combining multiple providers on one prompt. `src/index.ts` is the public entry point ŌĆö re-export the package's public API from there. The foundations (build, test, lint, type-check) are in place and three providers (Anthropic, OpenAI, Gemini) are implemented.

### Library structure (current)

- `src/types.ts` ŌĆö the provider-agnostic contract every provider implements: `Message`, `CompletionRequest`, `CompletionResult`, and `Provider` (`complete()` ŌåÆ full text, `stream()` ŌåÆ `AsyncIterable<string>` of text deltas).
- `src/providers/anthropic.ts` ŌĆö `AnthropicProvider`, talking to the Anthropic Messages API (`POST /v1/messages`) directly over the global `fetch` ŌĆö **no SDK dependency**. Default model `claude-opus-4-8`. Streaming parses the SSE body and yields `content_block_delta` ŌåÆ `text_delta` text. The request body is kept minimal (no `temperature`/`top_p`/`top_k`/`thinking` ŌĆö Opus 4.x rejects sampling params).
- `src/providers/openai.ts` ŌĆö `OpenAIProvider`, talking to the OpenAI Chat Completions API (`POST /v1/chat/completions`) directly over `fetch` ŌĆö **no SDK dependency**. Default model `gpt-4.1`. Auth via `Authorization: Bearer`. The optional `system` prompt is folded into a leading `{ role: "system" }` message (OpenAI has no top-level system field), and the token cap is sent as `max_completion_tokens` (works on the o-series too). Streaming parses SSE, yields `choices[0].delta.content`, and stops on the `data: [DONE]` sentinel.
- `src/providers/gemini.ts` ŌĆö `GeminiProvider`, talking to the Google Generative Language API (`POST /v1beta/models/{model}:generateContent`, streaming via `:streamGenerateContent?alt=sse`) directly over `fetch` ŌĆö **no SDK dependency**. Default model `gemini-2.5-pro`. Auth via the `x-goog-api-key` header. The model and action live in the URL path (not the body); the `assistant` role is mapped to Gemini's `model`, message text is wrapped as `parts: [{ text }]`, the optional `system` prompt is sent as a top-level `systemInstruction`, and the token cap is sent as `generationConfig.maxOutputTokens`. `complete()` concatenates `candidates[0].content.parts[].text` and reads the actual model from `modelVersion`; streaming parses SSE and yields the same text from each chunk. **Thinking-token gotcha:** Gemini 2.5 are thinking models and their thinking tokens count against `maxTokens` (`maxOutputTokens`), so a tiny cap can be fully consumed by thinking and return empty/truncated text (unlike Anthropic/OpenAI where `maxTokens` budgets only visible output). We deliberately do **not** disable thinking ŌĆö the default `gemini-2.5-pro` can't fully disable it ŌĆö so the live test uses a generous `maxTokens` (512) and the behavior is documented in the README rather than worked around in code.
- `src/registry.ts` ŌĆö `ProviderRegistry`, the package's **single point of access** to its providers. The provider classes are **not exported**; the registry constructs the built-ins by name from a config object. Constructor takes `ProviderRegistryConfig` (`{ anthropic?: AnthropicProviderOptions; openai?: OpenAIProviderOptions; gemini?: GeminiProviderOptions }`) and constructs only the providers present. `select(name)` is typed to `ProviderName` (`"anthropic" | "openai" | "gemini"`) and returns the `Provider` or throws an `Error` listing the configured names; plus `has(name: string)` and `names()` (fixed order: anthropic, openai, gemini). It never reads env vars ŌĆö keys come from the config. Also hosts `combine(request)` (see `src/combine/`): it resolves participant names ŌåÆ a roster of `{name, provider}` (each validated via `select()`), defaults the synthesizer to the first participant, and dispatches the strategy (one today, so it calls `consensus` directly with a comment marking the future seam ŌĆö a `switch` over the one-member `StrategyName` union trips `no-unnecessary-condition`).
- `src/combine/` ŌĆö the **combine** feature (step 4): make multiple providers cooperate on one prompt via a selectable strategy. **One file per strategy** (the user split this out so `combine` doesn't grow into one huge file). Public entry is `ProviderRegistry.combine(request)`.
  - `src/combine/index.ts` ŌĆö the **public types only** (no implementation): `StrategyName`; `CombineRequest` (extends `CompletionRequest` with `participants`/`synthesizer?`/`strategy?`/`attribution?`/`minParticipants?`); `ParticipantOutcome`; `CombineResult`. `src/index.ts` and `registry.ts` import these from `"./combine"` (resolves to this barrel). It's a **types-only** module (imports `ProviderName` from `../registry` as a `type`), so the combineŌåöregistry import cycle stays type-only/erasable ŌĆö keep it that way.
  - `src/combine/consensus.ts` ŌĆö the **consensus** strategy: the **internal** `consensus(roster, synthesizer, request)` (not exported from the package; `registry.ts` imports it from `"./combine/consensus"`) plus all its private helpers (`completionFor`, `composeSystem`, `runOutcome`, `renderAnswers`/`renderCritiques`/`renderConversation`, `synthesizerOrder`, the framing constants, `RosterEntry`/`Survivor` types). The synthesizer is passed **by name** (a `ProviderName`), not a positional index ŌĆö `synthesizerOrder` selects it by `name` (sound because `combine()` rejects duplicate participants). Phase 1 every participant drafts in parallel; phase 2 every survivor critiques all drafts (**anonymized** to `Answer A/B/C` by default; `attribution: "attributed"` shows provider names ŌĆö the returned result always keeps names regardless, since `ParticipantOutcome.provider` is set either way) and ends with a structured verdict (`BEST:` / `KEY FIX:` / `CONFIDENCE:` lines); phase 3 the synthesizer reads drafts + critiques and writes the final answer, framed to **adjudicate on correctness over popularity** (adopt a lone correct answer, don't average conflicting claims, don't favour its own ŌĆö now unidentifiable ŌĆö draft, surface unresolved disagreement) and to write a **self-contained answer that never alludes to the drafts/critiques/selection process or internal labels** (critical: with anonymization on, the model kept leaking `Answer A/B/C` and "the candidates" into the user-facing answer ŌĆö fixed in `SYNTH_FRAMING` by framing the drafts/critiques as _private input material_ and asking it to answer "as if for the first time", plus renaming the internal `## Candidate answers` body heading to `## Drafts` so the echoed word isn't in context). If you touch `SYNTH_FRAMING`, preserve these prohibitions. Prompt-only proved unreliable, so there is also a **sanitizing second pass** (`SANITIZE_FRAMING` + `sanitizeAnswer()`): after a non-empty synthesis, the chosen synthesizer is asked to rewrite the answer stripping any leftover process narration; it sees only the answer text (composed with the caller's `system` so voice/format carry over), and on failure/empty output returns the un-sanitized answer. This adds **one extra `complete()` call per combine** (only on the multi-provider synthesis path, not the single-provider degenerate path). Each phase is just a shaped `system` + single `user` message passed to `complete()` ŌĆö no provider-specific code, **no streaming for combine yet**. The draft and critique phases prepend a shared `CONCISE_DIRECTIVE` that strips ceremony (greetings/sign-offs/preamble) **but keeps reasoning/assumptions/caveats** ŌĆö the audience is other AIs that need the _why_ to critique it, so don't reword it to suppress reasoning; the user-facing synthesis is deliberately **not** constrained (don't add the directive there). `combine()` takes an optional `CombineOptions` (`{ onEvent? }`) threaded to `consensus(... , onEvent?)`; `consensus` emits `CombineEvent`s ŌĆö `{type:"phase"}` at each boundary and `{type:"draft"|"critique", provider, status}` as each call settles (emitted inside the per-item `Promise.all` map, so they fire as each resolves) ŌĆö via an internal `emit` that **swallows handler errors** (a progress listener must not break the run). Status events only ŌĆö **no token streaming** of the final answer; the result is still the returned `CombineResult` (no terminal event). Keep the framing constants free of each other's `phaseOf` markers ŌĆö the test's `phaseOf` switches on `"Rewrite the following"` (sanitize), `"lead assistant"` (synth), and `"produce only a critique"` (critique); the fake provider echoes its input for the sanitize phase. Fan-outs run via `Promise.all` over per-participant `runOutcome()` calls that capture success/failure as a `ParticipantOutcome`, so one provider failing never rejects the whole round. Survivor policy: 0 drafts ŌåÆ throw; a single-provider roster degrades to a plain completion; fewer than `minParticipants` (default 2) surviving drafts ŌåÆ throw; failed critiques are non-fatal (synthesize from drafts alone); a failed synthesizer falls back to the next survivor.
  - **Adding a strategy:** new `src/combine/<strategy>.ts` exporting its orchestrator, add a member to the `StrategyName` union in `index.ts`, and wire it into `registry.combine()` ŌĆö at the second member the direct call becomes a `switch (request.strategy ?? "consensus")` (lint-clean once the union has Ōēź2 members). Helpers in `consensus.ts` stay private to it; **only promote a shared helper when a second strategy actually reuses it** (the 2ŌĆō3x-duplication rule ŌĆö don't pre-extract a shared module for hypothetical needs). `CombineResult`'s `drafts`/`critiques` shape is consensus-specific; revisit it (likely a discriminated union on `strategy`) when a second strategy with different artifacts lands, not before.
  - Tests in `src/combine/__tests__/`: `consensus.test.ts` uses network-free fake `Provider` objects (no `fetch` mock) that classify the phase from the system prompt and record calls (it covers the `consensus()` strategy only ŌĆö the `ProviderRegistry.combine()` entry validation lives in `src/__tests__/registry.test.ts`, beside the code under test); the live `consensus.integration.test.ts` runs the real three-way consensus ŌĆö **triple-gated** on `RUN_LIVE_TESTS=1` + all three keys (it needs Ōēź2 participants); each provider uses its cheap model via the registry config default (combine applies one `model` to all participants, so per-provider models can't be set per call) and `maxTokens` is generous for Gemini's thinking tokens. **`noUncheckedIndexedAccess` gotcha:** don't zip parallel arrays by index (`a[i]` alongside `b[i]`) ŌĆö carry paired data in one structure (the roster) instead.
- New providers: add `src/providers/<name>.ts` implementing `Provider`, then wire it into `ProviderRegistry` (a new optional field on `ProviderRegistryConfig` + a construction branch). The provider class itself stays internal ŌĆö only `index.ts` re-exports its **options type** and the registry handles construction. `index.ts` is the single public entry point; do not export provider classes.
- Tests live in a `__tests__/` folder **colocated** with the code under test ŌĆö provider tests are in `src/providers/__tests__/` (import via `../<provider>`), combine tests are in `src/combine/__tests__/` (import the strategy via `../consensus`, and `src/`-level code via `../../registry` / `../../types`), and `src/`-level code like `registry.ts` is tested in `src/__tests__/` (import via `../registry`). The convention is one `__tests__/` per source folder, as close to the files under test as possible. Each provider has `<provider>.test.ts` (mocked unit tests) and `<provider>.integration.test.ts` (live, opt-in). Keep the `.test.ts` / `.integration.test.ts` suffix (the latter so the `test:integration` pattern filter matches them).

## Architecture / toolchain choices

- **Build = tsup, type-check = tsc.** tsconfig has `noEmit: true`; it exists for editor IntelliSense and `yarn typecheck` only. The build settings live in the `tsup` flags in `package.json`'s `build` script.
- **Tests run through `@swc/jest`** (`jest.config.mjs`), which transpiles TSŌåÆCJS fast and does _no_ type-checking ŌĆö types are covered separately by `yarn typecheck`. Test files import globals explicitly from `@jest/globals` (e.g. `import { describe, it, expect } from "@jest/globals"`) rather than relying on ambient types.
- **Dual-format packaging** is wired in `package.json` via the `exports` map (`import`ŌåÆ`.js`/`.d.ts`, `require`ŌåÆ`.cjs`/`.d.cts`). `files: ["dist"]` and `prepack: yarn build` mean the build is produced on publish/pack.
- **Node globals are typed via `@types/node`** (devDependency), referenced explicitly with `"types": ["node"]` in `tsconfig.json`. This is what makes `fetch`, `Response`, `ReadableStream`, `TextDecoder`, etc. resolve under the ES2022-only `lib` ŌĆö the provider transport relies on them.

## Package management ŌĆö Yarn 4 PnP

Uses **Yarn 4 (Berry) with Plug'n'Play** ŌĆö there is no `node_modules`; deps resolve via `.pnp.cjs` / `.pnp.loader.mjs`. Always use `yarn` (never `npm`).

- **PnP strictness gotcha:** you can only import packages declared directly in `package.json`. A transitive-only dep won't resolve (this is why `@jest/globals` is an explicit devDependency).
- `@jest/globals` versions lag `jest`'s ŌĆö they are not the same number (jest `30.4.2` Ōåö `@jest/globals` `30.4.1`). Match the version Jest actually pulls when bumping.
- `tsconfig` sets `ignoreDeprecations: "6.0"` solely because tsup's `.d.ts` bundler injects a now-deprecated `baseUrl` under TypeScript 6.

## Lint conventions (enforced)

`eslint.config.mjs` is a flat config: type-aware (`strictTypeChecked` + `stylisticTypeChecked`) + unicorn + promise for `**/*.ts`; Jest rules scoped to test files only; a non-type-checked block for `*.{js,cjs,mjs}`. Notable rules:

- **`type` over `interface`**; **`import type`** (inline style) for type-only imports; `consistent-type-exports`.
- Exported API must declare types (`explicit-module-boundary-types`) and functions need return types (`explicit-function-return-type`, callbacks exempt).
- `no-console: error` (a library shouldn't log to a consumer's console) ŌĆö off in tests.
- Prefix intentionally-unused args with `_`; unused imports are an error.
- Single-line `case` clauses omit braces. Prettier owns formatting (ESLint defers via `eslint-config-prettier`).
