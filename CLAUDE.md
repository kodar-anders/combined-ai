## What this is

A **plain TypeScript library** meant to be imported by other projects. Its purpose is to **combine several AI services/providers into one package** — selecting a single provider for a prompt, and (later) combining multiple providers on one prompt. `src/index.ts` is the public entry point — re-export the package's public API from there. The foundations (build, test, lint, type-check) are in place and three providers (Anthropic, OpenAI, Google) are implemented. (The Google provider speaks the Gemini API — its registry key/`name` is `"google"`, the company, consistent with `anthropic`/`openai`; the model/API it talks to is Gemini, so internal `toGemini*` helpers and the `gemini-2.5-pro` default model keep that name.)

### Library structure (current)

- `src/types.ts` — the provider-agnostic contract every provider implements: `Message`, `CompletionRequest`, `CompletionResult`, and `Provider` (`complete()` → full text, `stream()` → `AsyncIterable<string>` of text deltas). **`Message.content` is `string | ContentPart[]`** — a bare `string` is sugar for one text part (existing callers/tests untouched). `ContentPart` is a discriminated union: `TextPart` (`{type:"text", text}`), `ImagePart` (`{type:"image", source}`), `FilePart` (`{type:"file", source, filename?}` — image/file from Phase 1 multimodal), `ToolUsePart` (`{type:"tool_use", id?, name, input}` — replays an assistant tool call) and `ToolResultPart` (`{type:"tool_result", toolUseId?, name?, content, isError?}` — feeds a tool result back in a user message; Phase 4). The widening itself (Phase 0) was the one breaking change to `Message`; every member since just added to the union (additive). `MediaSource` = `{kind:"base64", mediaType, data} | {kind:"url", url, mediaType?}` (base64 carries its MIME type; `url`'s `mediaType` is optional — Gemini `fileData` uses it). Each provider maps content per-wire (string passthrough where the API allows it, else mapped parts — see provider bullets); **provider support varies and the mapper throws when unsupported** (e.g. OpenAI Chat Completions has no URL file source). When a future `ContentPart` member lands, the per-provider `to*Part`/`to*Block`/`to*Content` mappers and `textOf` in `combine/shared.ts` (which filters to text parts to render the prompt — combine renders to text, so non-text parts don't propagate) will fail to compile until handled (intended seam). `CompletionRequest.signal?: AbortSignal` is forwarded to every provider `fetch` (in the `init`) for timeout/cancellation; an aborted `fetch` rejects and `providerFetch` wraps it as a **transport** `ProviderError` (`cause` = abort reason). In combine, `completionFor` carries `signal` over (like `model`/`maxTokens`), so one signal threads into every phase/participant call and cancels the whole run. `CompletionResult` also carries `finishReason?: FinishReason` (normalized union `"stop" | "length" | "content_filter" | "tool_use" | "other"`), `rawFinishReason?: string` (the provider's exact value), and `refusal?: string` — `complete()`-only (no place to surface on `stream()`). Each provider maps its own stop field into the union (see provider bullets); a set `refusal` forces `finishReason` to `"content_filter"`. Lets callers distinguish truncation/refusal from a real empty answer instead of a bare `text: ""`. **Optional + only set when present**, so existing `toEqual({text, model})` tests still pass (Jest ignores `undefined` props). Threads through combine unchanged (`completionFor` passes the result through). New strategy/feature work that constructs a `CompletionResult` should keep these fields. `CompletionResult.usage?: Usage` (`{inputTokens, outputTokens, totalTokens}`) holds token usage — each provider's `extractUsage` parses its own field (Anthropic `usage.input_tokens/output_tokens`, no total ⇒ total=in+out; OpenAI `usage.prompt_tokens/completion_tokens/total_tokens`; Gemini `usageMetadata.promptTokenCount/candidatesTokenCount/totalTokenCount` — its total **includes thinking tokens** so it's kept verbatim, not in+out). Same optional/only-when-present rule; `complete()`-only. **Structured output:** `CompletionRequest.responseFormat?: ResponseFormat` (`{type:"json_schema", schema, name?}`) — a **raw JSON Schema** (no Zod/runtime dep; the user's locked choice). Each provider maps it to its native mechanism (Anthropic `output_config.format`, OpenAI `response_format.json_schema` with `strict:true`, Gemini `responseSchema` via `toGeminiSchema` — see provider bullets); applied in `#buildBody`, so it affects `complete()` **and** `stream()`. The model returns JSON in `text`; `complete()` also surfaces `CompletionResult.parsed?: unknown` (the shared `parseStructured` in `src/providers/structured.ts` = `JSON.parse(text)` when `responseFormat` set, else `undefined`; `undefined` too on invalid/truncated JSON — raw stays in `text`). **Cross-provider schema requirements** (driven by the strictest providers): every `object` sets `additionalProperties:false` + every prop in `required` with a single **non-null** type; avoid optional/nullable fields (OpenAI strict needs all-required; Gemini wants `nullable:true`, not a `["string","null"]` union or `anyOf`-null — `toGeminiSchema` does **not** translate null-unions, so they're not portable), recursive schemas, `$ref`, numeric/length constraints (Gemini ignores most keywords beyond types/enum/format). `complete()`-only for `parsed`; same optional/only-when-present rule. **Tool calling (Phase 4, single-provider):** `CompletionRequest.tools?: ToolDefinition[]` (`{name, description?, parameters: <JSON Schema>}`) + `toolChoice?: ToolChoice` (`"auto" | "any" | "none" | {name}`). On a tool call, `complete()` returns `CompletionResult.toolCalls?: ToolCall[]` (`{id?, name, input}` — `input` always an **object**: OpenAI's JSON-string `arguments` is parsed, Anthropic/Gemini are already objects) and `finishReason: "tool_use"` (Gemini reports `STOP`, so the provider overrides it when `functionCall` parts are present). Feed results back by replaying the conversation with `ToolUsePart` (assistant) + `ToolResultPart` (user) content parts; each provider maps them per-wire (see provider bullets). Surfaced by `complete()` only (not `stream()`); **`combine` rejects `tools`/`toolChoice`** (no multi-model tool loop — `registry.combine` throws, parallel to its `responseFormat` reject). Each provider's `to*ToolChoice` normalizes the choice vocabulary. **Cross-provider id/name requirements when replaying tool turns:** OpenAI **throws** if a `tool_use` lacks `id` or a `tool_result` lacks `toolUseId` (it requires them); Gemini **throws** if a `tool_result` lacks `name` (it matches by name); Anthropic **hoists** `tool_result` blocks to the front of the user turn (the API requires them first).
- `src/errors.ts` — the error **vocabulary**: `ProviderError` (exported) + the `apiError` helper (builds the `api` error from a non-2xx `Response`) all three providers use. `apiErrorFromBody(provider, status, data)` is the parallel for a **2xx body that carries `{ error }`** (status kept, e.g. 200) — each provider's `complete()` calls it when `isRecord(data.error)`, so a 200-with-error-body throws an `api` `ProviderError` instead of returning `text: ""`. `ProviderError` carries `provider` (`ProviderName`), a `kind` discriminant (`"api"` = error HTTP response → `status` set | `"transport"` = `fetch` itself rejected → `status` undefined, `cause` set), and `code`/`type` parsed from the body (`parseErrorBody`: string `error.code` → `code`; `error.type` or, for Gemini, `error.status` → `type`). Non-2xx message is lowercase `` `${provider} request failed (${status}): ${body}` `` (provider-test assertions follow). **Imports `ProviderName` from `./registry` as `type`-only** so providers→errors→registry has no runtime cycle — keep it type-only. **Mid-stream SSE `error` events + the "no body" case stay plain `Error`** (no HTTP status to attach), outside `ProviderError`'s `api`⇒`status` invariant. Also exports `aggregateError(message, causes: Error[])` — a feature-agnostic helper that returns an `AggregateError` (causes on `.errors`) when `causes` is non-empty, else a plain `Error`; combine's `noResultError` (in `combine/shared.ts`) is the thin adapter over it. Pure error layer — no `fetch`/transport code (that's `transport.ts`); imports nothing from `transport.ts` **or `combine/`** (so the generic helper, not `ParticipantOutcome`, lives here — keeps errors below the combine feature).
- `src/transport.ts` — the HTTP **transport** layer (how a request is made; the error vocabulary is `errors.ts`). Module-internal `providerFetch` wraps the global `fetch` and translates a rejection (network/DNS/abort) into a **transport** `ProviderError`. Exported `requestWithRetry(provider, input, init, retry?)` wraps it with bounded exponential backoff on `RETRYABLE_STATUSES` (429/503/529) for both `complete()`/`stream()`: returns the first non-retryable `Response` (caller still calls `apiError`), honors a `Retry-After` header (seconds or HTTP-date, capped at `MAX_BACKOFF_MS`=60s), and its `sleep` resolves early on `signal` abort so the next `fetch` rejects into the transport error. **Transport rejections are not retried** (scope = HTTP statuses). Config via the per-provider `retry?: RetryOptions` (`{maxRetries?, baseDelayMs?}`, exported from `index.ts`; defaults 2 / 500ms; `maxRetries: 0` disables) — each provider stores `#retry` and passes it through. Retry is per-`complete()` call, so combine benefits transparently (no combine changes). Imports `ProviderError` (value) from `./errors` + `ProviderName` (type) from `./registry`; `errors.ts` does **not** import back (one-way). Tested in `src/__tests__/transport.test.ts` (fake timers).
- `src/providers/anthropic.ts` — `AnthropicProvider`, talking to the Anthropic Messages API (`POST /v1/messages`) directly over the global `fetch` — **no SDK dependency**. Default model `claude-opus-4-8`. Messages map via `toAnthropicMessage` (a `string` content passes through; `ContentPart[]` → content blocks via `toAnthropicBlock`: text→`text`, image→`image`, file→`document`; `toAnthropicSource` → `{type:"base64", media_type, data}` or `{type:"url", url}`). Streaming consumes the shared `sseJson` iterator (see `src/providers/sse.ts`) and yields `content_block_delta` → `text_delta` text (returns on `message_stop`, throws on an `error` event). The request body is kept minimal (no `temperature`/`top_p`/`top_k`/`thinking` — Opus 4.x rejects sampling params). `finishReason` maps `stop_reason` (`end_turn`/`stop_sequence`→`stop`, `max_tokens`→`length`, `refusal`→`content_filter`, else `other`); `refusal` collects text from `type: "refusal"` content blocks (which `extractText` skips). Structured output → top-level `output_config: {format: {type:"json_schema", schema}}` (schema passed through as-is; no name/strict wrapper). Tools → `tools:[{name, description, input_schema}]` + `tool_choice` ({name}→`{type:"tool",name}`, else `{type:choice}`); `tool_use`/`tool_result` content blocks (`toAnthropicBlock`); `toAnthropicMessage` **hoists `tool_result` blocks first** in the user turn (the API requires it); `extractToolCalls` scans `tool_use` blocks; `stop_reason:"tool_use"`→`tool_use`.
- `src/providers/openai.ts` — `OpenAIProvider`, talking to the OpenAI Chat Completions API (`POST /v1/chat/completions`) directly over `fetch` — **no SDK dependency**. Default model `gpt-4.1`. Auth via `Authorization: Bearer`. `OpenAIProviderOptions.headers?` are spread into `#headers()` **last** (so they can override the defaults) — gateway auth/routing (OpenRouter `HTTP-Referer`/`X-Title`) + the openai-compatible custom-provider path. `OpenAIProvider`'s constructor takes a 2nd arg `name = "openai"` (an **internal** arg, not a public option — only the registry sets it) used as `this.name` **and** the `provider` passed to `requestWithRetry`/`apiError`/`apiErrorFromBody`, so an openai-compatible custom provider's errors attribute to its alias (the registry passes the registry key) instead of a hardcoded `"openai"`. The optional `system` prompt is folded into a leading `{ role: "system" }` message (OpenAI has no top-level system field), and the token cap is sent as `max_completion_tokens` (works on the o-series too). `toOpenAIMessages` sends a `string` content as-is; `ContentPart[]` → content parts via `toOpenAIPart`: text→`text`, image→`image_url` (base64 → a `data:` URI, else the url), file→`file` with `file_data` (base64 only — a **URL file source throws**, Chat Completions has none). Tools → `tools:[{type:"function", function:{name, description, parameters}}]` + `tool_choice` (`"any"`→`"required"`, {name}→`{type:"function",function:{name}}`, else passthrough). **`toOpenAIMessages` flatMaps** (one Message → several wire messages): an assistant message's `tool_use` parts become its `tool_calls` (`arguments` = `JSON.stringify(input)`); a user message's `tool_result` parts each become a separate `{role:"tool", tool_call_id, content}` message. a `tool_use` without `id` or a `tool_result` without `toolUseId` **throws** (OpenAI requires them). `extractToolCalls` reads `message.tool_calls` and **parses the `arguments` string** (`parseArguments`); `finish_reason:"tool_calls"/"function_call"`→`tool_use`. Streaming consumes the shared `sseJson` iterator and yields `choices[0].delta.content` (the `[DONE]` sentinel is skipped inside `sseJson`). `finishReason` maps `choices[0].finish_reason` (`stop`/`length`/`content_filter` pass through, `tool_calls`/`function_call`/else→`other`); `message.refusal` (when a non-empty string) is surfaced on `refusal` and overrides `finishReason` to `content_filter` even when `finish_reason` is `"stop"`. Structured output → `response_format: {type:"json_schema", json_schema:{name (defaults `"response"`), strict:true, schema}}`.
- `src/providers/google.ts` — `GoogleProvider` (registry key & `name` `"google"`), talking to the Google Generative Language API (`POST /v1beta/models/{model}:generateContent`, streaming via `:streamGenerateContent?alt=sse`) directly over `fetch` — **no SDK dependency**. Public options type `GoogleProviderOptions`. Default model `gemini-2.5-pro`. Auth via the `x-goog-api-key` header. Error-attribution strings (`requestWithRetry`/`apiError`/`apiErrorFromBody` + the stream-error `Error`s) use `"google"`/`"Google"`; the wire-mapping helpers (`toGemini*`), the `GeminiPart` type, the default model, and the API-semantic messages (e.g. the tool-result-needs-`name` throw) keep the `Gemini` name since they describe the Gemini API specifically. Non-obvious API shape: model + action live in the URL path (not the body); `assistant` role maps to `model`; the optional `system` prompt is a top-level `systemInstruction`; the token cap is `generationConfig.maxOutputTokens`. `toGeminiContent` wraps a `string` content as a single `{text}` part; `ContentPart[]` → parts via `toGeminiPart` (image and file map alike via `toGeminiMedia`: base64 → `inlineData {mimeType, data}`, url → `fileData {mimeType?, fileUri}` — `mimeType` omitted when the url source has none; `tool_use` → `functionCall {name, id?, args}`, `tool_result` → `functionResponse {name, id?, response:{result: content}}` — **throws if the tool_result has no `name`**, since Gemini matches results to calls by name). Tools → `tools:[{functionDeclarations:[{name, description, parameters}]}]` with **parameters run through `toGeminiSchema`** (same UPPERCASE-type transform as structured output) + `toolConfig.functionCallingConfig` ({name}→`{mode:"ANY", allowedFunctionNames:[name]}`, else `{mode: AUTO|ANY|NONE}`). `extractToolCalls` scans `functionCall` parts; finishReason stays `STOP`, so the provider **overrides** it to `tool_use` when tool calls are present — but **only on a clean stop** (normalized `stop`/undefined); a real `MAX_TOKENS`/`SAFETY` stop keeps its reason so a truncated tool call isn't masked. `extractText`/`extractToolCalls` share `firstCandidateParts`. **Gemini `fileData.fileUri` only accepts a Files API / `gs://` URI, not an arbitrary public web URL** (unlike Anthropic/OpenAI image URLs) — a plain `https://` url source is rejected by Gemini; documented on `toGeminiMedia`, not translated. **Thinking-token gotcha:** Gemini 2.5 are thinking models whose thinking tokens count against `maxOutputTokens` (unlike Anthropic/OpenAI, where `maxTokens` budgets only visible output), so a tiny cap can return empty/truncated text. We don't disable thinking (the default model can't fully disable it) — give Gemini a generous cap. `finishReason` maps `candidates[0].finishReason`, falling back to `promptFeedback.blockReason` when the prompt is blocked (no candidates): `STOP`→`stop`, `MAX_TOKENS`→`length`, `SAFETY`/`RECITATION`/`BLOCKLIST`/`PROHIBITED_CONTENT`/`SPII`→`content_filter`, else `other`. No `refusal` field (Gemini has no refusal-text field — the block reason lands in `rawFinishReason`). Structured output → `generationConfig.responseMimeType:"application/json"` + `responseSchema` built by `toGeminiSchema` (recursively UPPERCASEs `type` through `properties`/`items`/`anyOf`/`allOf`/`oneOf`; other keywords pass through; null-unions/`$ref`/constraints not translated).
- `src/registry.ts` — `ProviderRegistry`, the package's **single point of access** to its providers. The provider classes are **not exported**; the registry constructs the built-ins by name from a config object. Constructor takes `ProviderRegistryConfig` (`{ anthropic?; openai?; google?; custom? }`) and constructs only the providers present. **`custom?: Record<string, CustomProviderConfig>`** registers extra providers under caller-chosen names: `CustomProviderConfig` is a discriminated union — `{kind:"openai-compatible", apiKey, baseUrl, model, headers?, retry?}` (constructs an `OpenAIProvider` pointed at `baseUrl`, the escape hatch for OpenRouter/Together/Groq/Ollama/local — `baseUrl` excludes `/v1/chat/completions`, `model` required) | `{kind:"provider", provider}` (a `Provider` you bring yourself). Built via module-level `constructCustom(name, custom)` (passes `name` into `OpenAIProvider` so a gateway's errors attribute to its alias; has an exhaustive `default: never` like the combine switch); a custom name in `BUILT_IN_NAMES` (anthropic/openai/google) **throws** at construction. **`ProviderName` is `BuiltInProviderName | (string & Record<never,never>)`** (the `& {}` autocomplete trick — `Record<never,never>` not `{}` to satisfy `no-empty-object-type`), so custom names are accepted everywhere (`select`, combine participants, `Record<ProviderName,Usage>`) while built-ins still autocomplete. `select(name)` returns the `Provider` or throws an `Error` listing the configured names; plus `has(name: string)` and `names()` (built-ins in fixed order anthropic/openai/google, then custom names in definition order). It never reads env vars — keys come from the config. Also hosts `combine(request)` (see `src/combine/`): it **normalizes** each `ParticipantSpec` (module-level `normalizeParticipant` → `{id, providerName, model?, maxTokens?}`; bare string → id=name; object → id = `label ?? (model ? `${provider}-${model}` : provider)`; **throws on a falsy override** — empty `model` or non-positive/non-integer `maxTokens` — since `??` would otherwise forward `""`/`0` to the provider instead of falling back), validates the **shared** request shape (≥1 participant, **unique participant ids**, non-empty `messages`, known `strategy`), resolves the roster `{id, providerName, provider: select(providerName), model?, maxTokens?}`, and dispatches on `switch (request.strategy ?? "consensus")` with an exhaustiveness `default` that throws. **Consensus-only options** (`minParticipants` an integer in `[1, participants]`, `synthesizer` membership in the resolved **ids** — `#validateConsensusOptions(request, ids)`) are validated **inside the consensus branch** — _not_ in the shared preamble, so pipeline (which ignores them) never rejects a valid request over them. The consensus branch defaults the synthesizer to the first participant's **id**. **`responseFormat` is ensemble-only:** rejected (throws) for any non-ensemble strategy right after `strategy` is resolved (it would otherwise be silently dropped by `completionFor`), and in the `ensemble` branch it's **required** (throws if missing) **and must have an object root** (throws on an array/scalar `schema.type` — the field-wise vote needs named fields).
- `src/combine/` — the **combine** feature: make multiple providers cooperate on one prompt via a selectable strategy. **One file per strategy** (so `combine` doesn't grow into one huge file). Public entry is `ProviderRegistry.combine(request)`.
  - `src/combine/index.ts` — the public barrel: the combine types (`StrategyName`, `ParticipantSpec`, `CombineRequest`, `ParticipantOutcome`, `CombineResult` = `ConsensusResult | PipelineResult | EnsembleResult | BroadcastResult` discriminated on `strategy` (note `BroadcastResult` has **no `text`** — it returns every raw response, not one combined answer), `EnsembleAgreement`, `CombineUsage`, `CombineEvent` — the `{type:"response"}` variant is shared by **ensemble and broadcast**, `CombineOptions`) plus the one runtime value, `STRATEGY_NAMES` (`consensus`/`pipeline`/`ensemble`/`broadcast` — the source of truth `combine()` validates against). **Per-participant models:** `participants: ParticipantSpec[]` where `ParticipantSpec = ProviderName | {provider, model?, maxTokens?, label?}` — lets one combine mix models or reuse a provider with different models. Each participant has a unique **id** (label; defaults to provider name, or `<provider>-<model>` when model set); `synthesizer` is an **id** (`string`, not `ProviderName`). `ParticipantOutcome` carries both `id` (label) and `provider` (actual provider); `CombineEvent` variants likewise carry `id`+`provider`; `PipelineResult.finalParticipant` (renamed from `finalProvider`) is an id. All result members carry `usage?: CombineUsage` (`{total, byParticipant}` — `byParticipant` keyed by **id**) aggregating every model call the run made. It imports `ProviderName` from `../registry` as a **`type`**, so the combine↔registry cycle stays type-only/erasable — keep it that way (no value imports from `../registry`).
  - `src/combine/shared.ts` — helpers reused by **more than one** strategy: `completionFor`, `composeSystem`, `runOutcome`, `respondAll` (the verbatim-prompt parallel fan-out + `response`-event emit — shared by ensemble & broadcast; each strategy keeps its own accept/throw policy on the returned outcomes), `noResultError(message, outcomes)` (the "no usable result" throw, used by all four strategies' all-failed throws: the combine adapter over `errors.ts`'s `aggregateError` — extracts the failed outcomes' errors so the throw carries them on `.errors` (`AggregateError`), else a plain `Error`; message preserved so `toThrow` assertions hold), `renderConversation`, `sanitizeAnswer` (+ its private `SANITIZE_FRAMING`), `makeEmitter` (wraps `onEvent` into an error-swallowing `emit`), `aggregateUsage`/`outcomeUsage` (+ the `UsageEntry` type, keyed by `id`), and the `RosterEntry` type (`{id, providerName, provider, model?, maxTokens?}`). `runOutcome(id, providerName, run)` stamps both onto the outcome. `completionFor(request, system, messages, overrides?)` and `sanitizeAnswer(..., overrides?)` take a per-participant `ParticipantOverrides` (= `Pick<RosterEntry, "model" | "maxTokens">`) that **takes precedence over** the request-wide `model`/`maxTokens` (the fallback); since a `RosterEntry`/`Survivor`/`Running.entry` is structurally a valid `ParticipantOverrides`, strategies pass the **entry itself** (not a hand-built `{model, maxTokens}` literal). `sanitizeAnswer` returns `{text, usage?}` (not a bare string) so its billed call counts toward the aggregate; both strategies build `usage` by feeding all per-call usages (drafts/critiques/synth/sanitize, or stages/sanitize) through `aggregateUsage`. `completionFor` also carries `request.responseFormat` through (so ensemble's participant calls answer under the schema; consensus/pipeline reject `responseFormat` at the registry, so it's only ever set here for ensemble). A helper lands here only once a 2nd strategy reuses it (the 2–3x rule); strategy-private helpers stay in the strategy file.
  - `src/combine/consensus.ts` — the **consensus** strategy and its strategy-private helpers (`Survivor`, `LETTERS`, `renderAnswers`, `renderCritiques`, `synthesizerOrder`, `CONCISE_DIRECTIVE`, `CRITIQUE_FRAMING`, `SYNTH_FRAMING`); the cross-strategy helpers live in `shared.ts`. The internal `consensus(roster, synthesizer, request, onEvent?)` returns a `ConsensusResult` and is **not exported from the package**; `registry.ts` imports it from `"./combine/consensus"`. The synthesizer is passed **by id** (a participant label `string`), matched via `synthesizerOrder` on `s.id`; `Survivor = RosterEntry & {result}`, so it carries each participant's `id`/`providerName`/`model`/`maxTokens` into the critique/synth/sanitize calls. Each phase is a shaped `system` + single `user` message to `complete()` — no provider-specific code, **no streaming for combine yet**. Three phases:
    - **Phase 1 (draft):** every participant drafts in parallel.
    - **Phase 2 (critique):** every survivor critiques all drafts (**anonymized** to `Answer A/B/C` by default; `attribution: "attributed"` shows participant **ids** via `renderAnswers`/`renderCritiques` using `s.id`/`critique.id` — the returned `ParticipantOutcome` always carries `id`+`provider` regardless), ending with a structured verdict (`BEST:` / `KEY FIX:` / `CONFIDENCE:`).
    - **Phase 3 (synthesize):** the synthesizer reads drafts + critiques and writes the final answer.
    - **Framing invariants — preserve if you touch the constants** (rationale in the `consensus.ts` doc-comments): `SYNTH_FRAMING` makes the synthesizer adjudicate on **correctness over popularity** and write a **self-contained answer that never alludes to the drafts/critiques/process or internal labels** (the model kept leaking `Answer A/B/C` / "the candidates" — don't reintroduce that). Because prompt-only proved unreliable, a **sanitizing second pass** (`sanitizeAnswer()` + its `SANITIZE_FRAMING`, both in `shared.ts`) rewrites the synthesis to strip leftover process narration, falling back to the un-sanitized answer on empty/failure — this is **one extra `complete()` call** on the multi-provider path only. The draft/critique phases prepend `CONCISE_DIRECTIVE`, which strips ceremony **but keeps reasoning/assumptions/caveats** (the audience is other AIs that need the _why_) — don't reword it to suppress reasoning, and don't add it to the user-facing synthesis.
    - **Test coupling:** the test's `phaseOf` switches on literal markers in the framing constants — `"Rewrite the following"` (sanitize), `"lead assistant"` (synth), `"produce only a critique"` (critique). Keep each constant free of the others' markers.
    - **Events:** `combine()` accepts `CombineOptions` (`{ onEvent? }`) threaded to `consensus`, which emits `CombineEvent`s (`{type:"phase"}` per boundary; `{type:"draft"|"critique", provider, status}` as each call settles) via an `emit` that **swallows handler errors**. Status events only — no token streaming, no terminal event; the result is the returned `CombineResult`.
    - **Failure / survivor policy:** fan-outs use `Promise.all` over per-participant `runOutcome()` calls that capture success/failure as a `ParticipantOutcome`, so one provider failing never rejects the round. A successful draft only counts as a survivor if its text is non-empty (mirrors the synthesis empty-guard) — an empty `ok` draft is dropped so it can't satisfy `minParticipants` or render as a blank `### Answer`. 0 drafts → throw; single-provider roster → plain completion; fewer than `minParticipants` (default 2) surviving drafts → throw; failed critiques are non-fatal (synthesize from drafts alone); a failed synthesizer falls back to the next survivor.
  - `src/combine/pipeline.ts` — the **pipeline** strategy (sequential refinement / conveyor belt). Internal `pipeline(roster, request, onEvent?)` returns a `PipelineResult`; `registry.ts` imports it from `"./combine/pipeline"`. Participant **order is the conveyor order**. Loops the roster keeping a running answer: the first stage with no running answer yet uses `PIPELINE_FIRST_FRAMING`, later stages refine the current answer under `PIPELINE_REFINE_FRAMING` (a `## Question` + `## Current answer` body). A stage **advances** the running answer only if it's `ok` **and** non-empty; failed/empty stages are recorded in `stages` and the previous answer carries forward. Final answer = last advancing stage's text/model + its `id` as `finalParticipant`. Throws only if **no** stage advanced. **No `minParticipants`/`synthesizer`/`attribution`** (consensus-only — ignored). Emits `{type:"stage", id, provider, status, index}` per stage. `Running = {entry: RosterEntry, text, model, needsSanitize}` (carries the producing stage's entry so its overrides reach the sanitize call). **Sanitize** (`sanitizeAnswer`, shared with consensus) runs only when a refining stage actually **changed** the text (tracked via `Running.needsSanitize`) — a lone first-stage answer or a verbatim passthrough is returned raw, so the extra call only fires when it can matter. **Framing rationale (preserve if you touch it):** refiners are told to treat the current answer as a strong baseline and preserve correct content (no downstream synthesizer catches a regression). **Test coupling:** `pipeline.test.ts`'s `phaseOf` keys on `"Rewrite the following"` (sanitize, shared), `"revise the current answer"` (refine), else first — keep each framing constant free of the others' markers and of consensus's (`"lead assistant"`, `"produce only a critique"`).
  - `src/combine/ensemble.ts` — the **ensemble** strategy (multi-model vote on structured output — the differentiator). Internal `ensemble(roster, request, onEvent?)` returns an `EnsembleResult`; `registry.ts` imports it from `"./combine/ensemble"`. Every participant answers the prompt **in parallel** under the same `request.responseFormat` (via the shared `respondAll`), each returning a `parsed` object. Only `ok` responses whose `parsed` is a **plain object** count (a failed call, empty/invalid JSON → `parsed` undefined, or a non-object top-level value are dropped); throws if **none** qualify. **Mechanical merge** (`mergeObjects`/`mergeField`, no LLM): field-wise over the first-seen union of top-level keys, **every field by majority vote** — mode (most common by `stableKey` deep-equality; strict `>` so first-seen wins ties), so the merged value is always one a model actually returned (no synthesized/averaged values; an earlier median was dropped — see the plan doc — because it fabricated non-schema values and desynced from `agreement`). Per-field `agreement` = `maxCount / objects.length` — **denominator is all valid responses**, not just the ones that returned the field, so a field most models omitted scores low (honest confidence). `overall` = mean of per-field. Result carries `merged` (object), `text` (=`JSON.stringify(merged)`), `agreement` (`{overall, byField}`), `responses` (all outcomes, participant order), `usage`. Emits `{type:"response", id, provider, status}` per settle. **Object-root only:** `registry.combine` rejects an array/scalar-root schema for ensemble up front (the vote needs named fields). **Nested-merge is intentionally shallow** (nested objects/arrays voted as whole values, not recursively merged). **`toSorted` gotcha:** the ES2022 lib lacks it, so `stableKey` uses `.sort()` with an `eslint-disable unicorn/no-array-sort`.
  - `src/combine/broadcast.ts` — the **broadcast** strategy (fan-out, **no combine**). Internal `broadcast(roster, request, onEvent?)` returns a `BroadcastResult`; `registry.ts` imports it from `"./combine/broadcast"`. Every participant answers the prompt **in parallel** via the shared `respondAll` (verbatim `system`+`messages`, no shaped framing — the point is the unmodified per-model answer; no `composeSystem`/`sanitizeAnswer`). Returns `{strategy, responses, usage?}` — **no `text`** (there's no single answer). Throws (`noResultError`) **only when zero** participants succeed; an `ok` with empty text still counts (unlike consensus/pipeline, which drop empty answers because they can't advance). Reuses the `{type:"response"}` event. `registry.combine` rejects `responseFormat` for broadcast (it's `!== "ensemble"`) — structured output is out of scope by design.
  - **Adding a strategy:** new `src/combine/<strategy>.ts` exporting its orchestrator (returning its own `…Result` member), add a member to the `STRATEGY_NAMES`/`StrategyName` union in `index.ts`, add a `…Result` to the `CombineResult` union (discriminated on `strategy`), and add a `case` to the `switch (request.strategy ?? "consensus")` in `registry.combine()`. Promote a helper to `shared.ts` **only when the new strategy actually reuses it** (the 2–3x-duplication rule); strategy-private helpers stay in the strategy file. If the strategy emits different progress events, add a `CombineEvent` variant.
  - Tests in `src/combine/__tests__/`: `consensus.test.ts`, `pipeline.test.ts`, `ensemble.test.ts`, and `broadcast.test.ts` each use network-free fake `Provider` objects (no `fetch` mock) — consensus/pipeline classify the phase from the system prompt (each test has its own `phaseOf`); ensemble's fake returns a fixed `parsed` object (or throws) and records calls (to assert the schema threaded through); broadcast's fake records calls (to assert the **raw** prompt is sent unframed). They cover the `consensus()` / `pipeline()` / `ensemble()` / `broadcast()` orchestrators directly. The `ProviderRegistry.combine()` entry validation lives in `src/__tests__/registry.test.ts`, beside the code under test — including the ensemble-requires-`responseFormat` / non-ensemble-reject-`responseFormat` throws (consensus/pipeline/broadcast), the **duplicate-label** throw, an **end-to-end per-participant-model** test that runs the same BYO `Provider` twice with different models through `combine(strategy:"pipeline")` and asserts each stage's `request.model`, and a **broadcast dispatch** test (the places registry dispatch is exercised). The live `consensus.integration.test.ts` / `pipeline.integration.test.ts` / `ensemble.integration.test.ts` / `broadcast.integration.test.ts` run the real three-way flow — **triple-gated** on `RUN_LIVE_TESTS=1` + all three keys; each participant is a bare provider name using its cheap model via the registry config default (per-participant `model` overrides are an option now, but the live tests rely on config defaults) and `maxTokens` is generous for Gemini's thinking tokens. Both integration tests narrow the returned `CombineResult` on `result.strategy` before touching strategy-specific fields. **`noUncheckedIndexedAccess` gotcha:** don't zip parallel arrays by index (`a[i]` alongside `b[i]`) — carry paired data in one structure (the roster / the running answer) instead.
- New providers: add `src/providers/<name>.ts` implementing `Provider`, then wire it into `ProviderRegistry` (a new optional field on `ProviderRegistryConfig` + a construction branch). The provider class itself stays internal — only `index.ts` re-exports its **options type** and the registry handles construction. `index.ts` is the single public entry point; do not export provider classes. **Provider-shared helpers** live in their own small modules: `src/providers/extract.ts` (`extractModel(data, fallback, field?)` — `field` defaults to `"model"`; Gemini passes `"modelVersion"` — plus the **exported `isRecord`** that the SSE module and all three providers import), `src/providers/structured.ts` (`parseStructured`), and `src/providers/sse.ts` (`sseJson(body)` — the shared SSE reader: an async generator that yields each `data:` line's parsed JSON object, skipping blank/non-JSON/non-object lines, **ending the stream on the `[DONE]` sentinel** so a server holding the connection open past it can't hang the consumer, flushing a final unterminated line, and always releasing the reader; each `stream()` just `for await`s it and handles provider-specific events/errors). All three `stream()` methods share this; the per-provider streaming detail above is just the event handling, not the parsing. `toArray` stays duplicated per-provider (trivial, 3 copies — convention); `isRecord` was promoted to `extract.ts` once the shared modules also needed it (it lived in 6 files). `errors.ts`/`transport.ts` keep their own `isRecord` (lower layer, no `providers/` dependency).
- Tests live in a `__tests__/` folder **colocated** with the code under test — provider tests are in `src/providers/__tests__/` (import via `../<provider>`), combine tests are in `src/combine/__tests__/` (import the strategy via `../consensus`, and `src/`-level code via `../../registry` / `../../types`), and `src/`-level code like `registry.ts` is tested in `src/__tests__/` (import via `../registry`). The convention is one `__tests__/` per source folder, as close to the files under test as possible. Each provider has `<provider>.test.ts` (mocked unit tests) and `<provider>.integration.test.ts` (live, opt-in). Keep the `.test.ts` / `.integration.test.ts` suffix (the latter so the `test:integration` pattern filter matches them).

## Commands

- `yarn build` — bundle `src/index.ts` to `dist/` via **tsup** as dual ESM (`index.js`) + CJS (`index.cjs`) with `.d.ts`/`.d.cts` declarations and sourcemaps.
- `yarn typecheck` — `tsc --noEmit`. tsc is type-check only; tsup owns the build.
- `yarn test` — Jest. Single test: `yarn test <pattern>` or `yarn test -t "<name>"`. Watch: `yarn test:watch`.
- `yarn test:integration` — **live** API tests (`*.integration.test.ts`). Double-gated per provider: a suite runs only when both `RUN_LIVE_TESTS=1` (the script sets it) and that provider's key are present (`ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY`); otherwise it's `describe.skip`, so plain `yarn test` always skips it and never makes a paid call. Live tests use the cheap model per provider (`claude-haiku-4-5` / `gpt-4.1-mini` / `gemini-2.5-flash`) with a small `maxTokens` to keep cost negligible — single-call suites use ~16 (Gemini 512, to leave room for its thinking tokens), the combine suites (consensus, pipeline) use 2048. The script defaults to all integration tests; append a Jest filename pattern to narrow to one — `yarn test:integration openai.integration` (the pattern **replaces** the `integration` default).
  - **Env loading:** `jest.setup.cjs` (wired via `setupFiles` in `jest.config.mjs`) loads a gitignored `.env` via `dotenv` so the key is picked up automatically. Copy `.env.example` → `.env` and paste your `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY`. **Only the tests read env** — the library itself never reads env vars; consumers always pass `apiKey` to the provider explicitly.
- `yarn lint` / `yarn lint:fix` — ESLint.
- `yarn format` / `yarn format:check` — Prettier.

## Architecture / toolchain choices

- **Build = tsup, type-check = tsc.** tsconfig has `noEmit: true`; it exists for editor IntelliSense and `yarn typecheck` only. The build settings live in the `tsup` flags in `package.json`'s `build` script.
- **Tests run through `@swc/jest`** (`jest.config.mjs`), which transpiles TS→CJS fast and does _no_ type-checking — types are covered separately by `yarn typecheck`. Test files import globals explicitly from `@jest/globals` (e.g. `import { describe, it, expect } from "@jest/globals"`) rather than relying on ambient types.
- **Dual-format packaging** is wired in `package.json` via the `exports` map (`import`→`.js`/`.d.ts`, `require`→`.cjs`/`.d.cts`). `files: ["dist"]` and `prepack: yarn build` mean the build is produced on publish/pack.
- **Node globals are typed via `@types/node`** (devDependency), referenced explicitly with `"types": ["node"]` in `tsconfig.json`. This is what makes `fetch`, `Response`, `ReadableStream`, `TextDecoder`, etc. resolve under the ES2022-only `lib` — the provider transport relies on them.

## Package management — Yarn 4 PnP

Uses **Yarn 4 (Berry) with Plug'n'Play** — there is no `node_modules`; deps resolve via `.pnp.cjs` / `.pnp.loader.mjs`. Always use `yarn` (never `npm`).

- **PnP strictness gotcha:** you can only import packages declared directly in `package.json`. A transitive-only dep won't resolve (this is why `@jest/globals` is an explicit devDependency).
- `@jest/globals` versions lag `jest`'s — they are not the same number. Match the version Jest actually pulls when bumping.
- `tsconfig` sets `ignoreDeprecations: "6.0"` solely because tsup's `.d.ts` bundler injects a now-deprecated `baseUrl` under TypeScript 6.

## Lint conventions (enforced)

`eslint.config.mjs` is a flat config: type-aware (`strictTypeChecked` + `stylisticTypeChecked`) + unicorn + promise for `**/*.ts`; Jest rules scoped to test files only; a non-type-checked block for `*.{js,cjs,mjs}`. Notable rules:

- **`type` over `interface`**; **`import type`** (inline style) for type-only imports; `consistent-type-exports`.
- Exported API must declare types (`explicit-module-boundary-types`) and functions need return types (`explicit-function-return-type`, callbacks exempt).
- `no-console: error` (a library shouldn't log to a consumer's console) — off in tests.
- Prefix intentionally-unused args with `_`; unused imports are an error.
- Single-line `case` clauses omit braces. Prettier owns formatting (ESLint defers via `eslint-config-prettier`).
